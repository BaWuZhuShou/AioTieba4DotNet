# Public API coverage matrix

This document is the canonical maintained inventory for the public v3 consumer surface. The source of truth is the public declarations in `AioTieba4DotNet/Clients/TiebaClient.cs`, `AioTieba4DotNet/Clients/ITiebaClient.cs`, `AioTieba4DotNet/Clients/ITiebaClientFactory.cs`, `AioTieba4DotNet/Clients/TiebaClientFactory.cs`, `AioTieba4DotNet/DependencyInjection.cs`, and the six module contracts under `AioTieba4DotNet/Contracts/`. `docs/reference/modules.md` is a consumer-facing cross-check only; it is not the canonical inventory source.

Internal `AioTieba4DotNet/Api/*` request families and generated files under `AioTieba4DotNet/Api/**/Protobuf/*` are **non-targets** for direct coverage claims. Public coverage rows track only consumer-facing entrypoints, root module accessors, factory/DI surfaces, and the public module contracts.

This matrix stays aligned with `docs/related/parity.md` through the retained artifact model. `pwsh ./scripts/verify-local.ps1 -ValidateOnly` must see the active docs tuple together with `.sisyphus/evidence/parity-truth-freeze.json`, `.sisyphus/evidence/parity-gap-ledger.json`, `.sisyphus/evidence/local-verification.manifest.json`, and `.sisyphus/evidence/local-verification.manifest.schema.json`. `pwsh ./scripts/test-lane.ps1 sequence-dry-run` remains a wrapper-only route printout for `Suite:SafeOrdered` and `Suite:RestrictedOrdered`.

## Verification command shorthands

- `build` = `dotnet build AioTieba4DotNet.sln --configuration Release --no-restore`
- `direct-api-safe(<category>)` = `dotnet test AioTieba4DotNet.Tests.Online/AioTieba4DotNet.Tests.Online.csproj --configuration Release --nologo --filter "TestCategory=<category>&TestCategory=Tier:Safe"`
- `direct-api-restricted(<category>)` = `dotnet test AioTieba4DotNet.Tests.Online/AioTieba4DotNet.Tests.Online.csproj --configuration Release --nologo --filter "TestCategory=<category>&TestCategory=Tier:Restricted"`

## Disposition legend

`Current coverage` is the evidence snapshot for this exact row today. `Disposition` is the plan-compliant execution disposition and must not be read as a covered/gap flag.

- `Safe online behavioral`: the row belongs in the default `Tier:Safe` online reality once direct coverage exists.
- `Restricted online behavioral`: the row belongs in the explicit opt-in `Tier:Restricted` online reality once direct coverage exists.
- `Offline contract/unit`: the row should be proven without live environment dependence.
- `Deferred with rationale`: the row does not have a plan-compliant executable path in the current suite reality because a concrete blocker/risk exists now.

## Lane execution rules

- `Target lane = offline contract/unit`: no live environment is required; permission assumptions and fail-closed gating are not applicable.
- `Target lane = safe`: use `Tier:Safe`. No restricted capability is assumed. Authenticated or messaging rows require dedicated safe credentials when the public behavior needs them, and all dedicated safe assets must fail closed with `Assert.Inconclusive(...)` instead of falling back to public forums, users, or chats.
- `Target lane = restricted`: use `Tier:Restricted`. The row assumes explicit restricted `optIn`, dedicated restricted credentials, the named capability (`Moderation` or `Admin`), and dedicated restricted assets. Missing opt-in, credentials, capability, or assets must stop the scenario before any guarded mutation body executes.
- `Target lane` records the intended lane. `Disposition` records whether that lane is active now or deferred by a concrete blocker.

## Direct `Api:*` eligibility rule

- Public online behavior methods on `IClientModule`, `IForumModule`, `IThreadModule`, `IUserModule`, `IMessagesModule`, and `IAdminModule` are expected to gain direct `Api:<Module>.<Method>` categorization once row-level online coverage exists, even when the current taxonomy is intentionally narrower or family-level.
- Overload families may continue sharing a single `Api:<Module>.<Method>` category, but row-level evidence still has to call out which overload has direct proof today and which overloads still only have sibling-family evidence.
- A row is first-class filterable only when its verification command advertises `direct-api-safe(...)` or `direct-api-restricted(...)`. Rows that remain `build`-only are intentionally not discoverable through first-class `Api:*` filtering.
- Root constructors, root accessors, `Dispose()`, factory/DI entrypoints, default interface properties, and offline parsing helpers are **not** direct `Api:*` targets.
- Deferred rows must not claim first-class `Api:*` filterability until the blocker named in their rationale is resolved and a direct scenario is approved.

## Root-surface representation rule

- This matrix records exactly one row per public member **or** per intentionally shared public behavior surface.
- `TiebaClient` constructors are listed as their own rows because each constructor is a distinct public entrypoint.
- `TiebaClientFactory` constructor and public `CreateClient(...)` overloads are listed as their own rows because the concrete type is itself a public entrypoint surface, and the concrete overloads carry implementation-visible behavior such as BDUSS validation and public delegation paths.
- `TiebaClient.Forums`, `TiebaClient.Threads`, `TiebaClient.Users`, `TiebaClient.Admins`, `TiebaClient.Messages`, `TiebaClient.Client`, and `TiebaClient.Dispose()` are intentionally represented once by the corresponding `ITiebaClient` behavior-surface rows below. In source, those concrete members are pure `inheritdoc` mirrors of the same public contract shape and do not add concrete-only parameters, return types, or alternate behavior beyond satisfying `ITiebaClient`.
